# Frontend Implementation Plan: Agentic Penetration Testing System

This document outlines the detailed implementation plan for the frontend of the Agentic Penetration Testing system. It specifies the architecture, user interface flows, and visual design requirements based on the presentation aesthetics.

## 1. Technical Architecture

*   **Frontend Technologies**: Vanilla HTML, CSS, and JavaScript. This approach ensures a lightweight, fully self-contained interface without full Node.js project overhead.
*   **Backend Bridge**: A lightweight Python web server (e.g., FastAPI or Flask) to bridge the browser with the local agentic system loop.
    *   **Responsibilities**: Serve the static frontend assets, expose an endpoint to trigger `main.py` via an asynchronous process, and provide a WebSocket or Server-Sent Events (SSE) route to aggressively stream standard output (`stdout`) to the frontend in real-time.

## 2. Design System & Aesthetics

The application must strictly follow the "hacker-chic" dark mode visual identity established in `pentest_presentation_v2.html`.

*   **Color Palette (CSS Variables)**:
    *   `--green: #00ff88;` (Primary accent, interactive elements, success states)
    *   `--green-dim: #00cc6a;` (Secondary accent)
    *   `--red: #ff3b5c;` (Errors, failed states, warnings)
    *   `--yellow: #ffd166;` (Warnings, cautions)
    *   `--blue: #00b4ff;` (Information, neutral statuses)
    *   `--bg: #050a06;` (Primary dark background)
    *   `--bg2: #080f09;` (Secondary dark background for panels)
    *   `--text: #c8f0d8;` (Primary text color)
    *   `--text-dim: #6a9474;` (Muted/secondary text)
*   **Typography**:
    *   `Share Tech Mono`, `monospace`: Used for terminal logs, data arrays, and technical labels.
    *   `Orbitron`, `sans-serif`: Used for primary headers, titles, and prominent numbers.
    *   `Inter`, `sans-serif`: Used for readable general body copy.
*   **Visual Effects**:
    *   Implement CSS-based CRT scanlines and a static noise SVG overlay.
    *   Utilize glassmorphism techniques for layered panels with subtle `--green` block-shadows indicating depth and focus.

## 3. Application Views & User Flow

The interface consists of three primary screens. The transition between these screens should be handled fluidly using CSS transforms (e.g., sliding off-screen).

### View 1: Landing Screen (Configuration)
The initial entry point, focusing on target configuration and authorization.
*   **Layout**: Clean, vertically centered layout.
*   **Core Inputs**:
    *   A large text input for the Target URL.
    *   A dropdown to select the Scan Mode (`Active` vs `Passive`).
*   **Advanced Configuration Panel (Collapsible)**:
    *   Inputs to define specific Scope Restrictions.
    *   Checkboxes for OWASP category focuses (if applicable).
*   **Authorization**:
    *   A prominent checkbox stating: "I confirm I am authorized to test this target."
*   **Primary Action**:
    *   A massive "Start Scan" button.
    *   **Constraint**: This button must securely remain disabled until a valid URL is detected and the authorization checkbox is marked.
*   **Transition**: Clicking Start initiates the backend system call and transitions the view structurally into the Pipeline.

### View 2: Pipeline View (Live Execution)
The execution monitoring dashboard.
*   **Header Bar**: Displays the configured Target URL and an active session timer.
*   **Sidebar (Agents List)**:
    *   Lists the 6 core agents chronologically: Reconnaissance, Authentication, Mapping & Crawl, Vulnerability Analysis, Exploitation, Reporting.
    *   Uses dynamic status styling. The active agent pulsates or glows (using the `--green` palette), while completed agents receive a checkmark, and pending agents remain muted. This state is inferred dynamically by parsing the live log stream.
*   **Main Stage (Live Log viewer)**:
    *   A terminal-emulated div.
    *   Automatically pins scroll to the bottom as new WebSocket text events arrive.
    *   Implements basic regex-based syntax highlighting locally (e.g., coloring lines containing "[ERROR]" with `--red`).

### View 3: Report View (Results)
Displays the final artifacts generated by the system.
*   **Trigger**: The backend broadcasts a termination signal over the socket when `main.py` concludes, automatically sliding the view to the Report Screen.
*   **Header Bar**: Replaces the timer with a "New Scan" button, resetting the application state loop.
*   **Report Display**:
    *   Fetches the final generated Markdown file from the `output/reports/` directory.
    *   Parses the markdown clientside (e.g., via `marked.js`) and renders it cleanly using the system's typography.
    *   Provides functional "Export to PDF" and "Print" actions.

## 4. Execution Phases

1.  **Phase 1: Backend Scaffolding**: Create the asynchronous python script to wrap `main.py` and handle the WebSocket log streaming architecture.
2.  **Phase 2: UI Foundation**: Setup the `index.html` structure and `index.css` global styles incorporating the defined presentation variables.
3.  **Phase 3: JavaScript Implementation**: Write the client-side logic handling form validation, view transition mechanics, and WebSocket message processing.
4.  **Phase 4: Output Rendering**: Integrate markdown parsing for the final report display and polish all CSS micro-animations.
